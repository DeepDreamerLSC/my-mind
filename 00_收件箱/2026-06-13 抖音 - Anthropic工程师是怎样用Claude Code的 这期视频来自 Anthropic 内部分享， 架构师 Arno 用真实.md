Anthropic工程师是怎样用Claude Code的? 这期视频来自 Anthropic 内部分享，
架构师 Arno 用真实项目演示了三个关键工作流。 
第一个，让 Claude 先采访你。 
别急着写需求，
告诉 Claude 领域和受众，
让它像产品经理一样追问。 
"这是给朋友用的工具，还是有第二类用户？" 
需求从一次性说明，
变成一轮轮真实访谈。 
第二个，用 HTML 规格稿替代长 Markdown。 
Markdown 超过两百行就很难认真读完。 
HTML 可以把信息压进可看、可点、可截图的产物。 
团队直接点击、比较、反馈，
不用靠想象理解文字。 
第三个，把验证嵌进产物本身。 
普通测试关注代码能否通过，
验证关注产物是否能被人和 Agent 原生检查。 
组件把状态发布到 DOM，
Agent 可以直接读取数据合同运行验证。 
更重要的是，
同一套验证要给三个对象用： 
人看的 dashboard，
Agent 从浏览器驱动的方式，
CI 里 headless 跑的命令。 
三者围绕同一份 manifest 工作。 
人在页面上能看见的失败，
Agent 也能读到。 
Arno 还演示了验证录制功能。 
验证步骤可以被录成 clips，
打包下载，或者分享给同事。 
这份证据包会改变 code review 的颗粒度。 
Reviewer 不必只看 diff，
可以看到 Agent 怎么跑、哪里失败、怎样修复。 
一句话总结： 
会用 Claude Code 的团队，
差距会慢慢从"谁更会写 prompt"，
转向"谁更会设计 Agent 可以工作的环境"。 
下次让 Agent 做长任务前，先问三件事： 
需求有没有被采访清楚？
规格能不能被人快速看懂？
结果能不能被 Agent 自己验证？ 
先把这三件事补齐，再谈规模化。

### 摘要

这期视频整理的是 Anthropic 内部一次关于 Claude Code 的分享。Arno 用真实项目演示了三件事：先让 Claude 像产品经理一样采访需求；用可点击、可截图的 HTML 规格稿替代超长 Markdown；把验证嵌进产物本身，让人、Agent 和 CI 围绕同一份 manifest 协作。核心结论是，团队竞争力会从“谁更会写 prompt”转向“谁更会设计 Agent 可以工作的环境”。

### 关键点

- 先让 Claude 采访需求，而不是一开始就硬写完整规格。
- 用 HTML 规格稿替代超长 Markdown，更适合点击、截图和团队反馈。
- 验证关注的不只是代码通过，还包括产物能否被人和 Agent 原生检查。
- 组件状态应直接暴露到 DOM，方便 Agent 读取并执行验证。
- 同一套验证最好同时服务于 dashboard、浏览器驱动 Agent 和 CI headless 命令。
- 验证步骤可以录制成 clips，作为 code review 的补充证据。
- 团队差距会逐渐从 prompt 技巧转向 Agent 工作环境设计能力。
- 在让 Agent 执行长任务前，先确认需求是否采访清楚、规格是否易读、结果是否可自验证。
