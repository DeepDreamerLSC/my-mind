## 适用场景

- 需要让 Codex 从零开始接手一个新目录或新项目。
- 需要先跑通最小版本，再逐步加入 UI、调试、Git、插件和自动化。
- 需要在安全、效率和权限之间明确选择工作模式。
- 需要把一次任务沉淀为后续可复用的流程，而不是停留在单次对话。

## 提示词

请把当前任务当成一次从 0 到 1 的 Codex 落地流程，而不只是一次单点实现。

按下面顺序推进，并在每一步给出简短结论：

1. 明确目标
   说明这次要交付什么、工作目录是什么、验收标准是什么。
2. 检查环境
   判断是否已有项目目录、依赖、运行方式、测试方式和 Git 仓库。
3. 选择权限与执行模式
   明确当前更适合保守模式、审查模式，还是全自动模式；说明理由和风险。
4. 先做最小可运行版本
   优先交付一个能运行、能验证的最小版本，不先追求完整美化。
5. 立即验证
   用最小代价验证功能是否真的可用；如果预览环境有限制，要区分“环境限制”和“真实缺陷”。
6. 迭代修正
   把后续需求拆成小步：界面调整、局部修改、Bug 排查、预览验证。
7. 留下版本锚点
   在合适的时候建议建立 Git 检查点，避免后续修改失控。
8. 判断是否需要插件或额外能力
   如果任务依赖浏览器、桌面应用、演示文稿、图片生成或自动化，再明确调用相应能力。
9. 判断是否值得沉淀
   识别这次流程里哪些步骤值得沉淀为 skill、提示词、脚本或自动化。
10. 最后输出
   给出：
   - 当前已完成内容
   - 验证证据
   - 剩余风险
   - 下一步最值得做的一件事

约束：

- 不要一开始就做大而全方案，先跑通最小闭环。
- 不要把预览器限制误判为业务 Bug。
- 不要自动提交、推送或做破坏性操作，除非我明确要求。
- 如果发现可以复用的流程，先总结为候选，不直接声称为长期最佳实践。
- 涉及产品价格、模型、权限策略或平台功能时，优先提示我做事实复核。
