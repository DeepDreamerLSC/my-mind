## 高频重复动作

1. 收件箱录入
   - 用户不断丢入 YouTube、抖音、小红书等链接。
   - Codex 需要解析元数据、公开文案、转写或字幕，并写入 `00_收件箱/`。
   - 已沉淀为 `inbox-capture`。

2. 收件箱巡检
   - 每次入箱后都需要判断是否值得沉淀、建议去向和风险。
   - 已沉淀为 `inbox-triage`。

3. 候选沉淀
   - 巡检建议之后，需要把高价值条目生成提示词库、资料库、项目任务或原子笔记候选。
   - 当前只手动完成了一次：`一段提示词，让 Codex 越用越聪明` -> `75_提示词库/Codex工作流/Codex 反向沉淀 Skill.md`。
   - 这是下一步最值得自动化的环节。

4. 来源状态回写
   - 生成候选草稿后，需要把来源笔记从 `待分拣` 改为 `已分拣`，并追加 `沉淀记录` 回链。
   - 当前规则已写入设计稿，但还没有脚本化。

5. 转写质量检查
   - 抖音、小红书短视频经常依赖 `tiny` 转写，英文术语容易误识别。
   - 已有术语表修正，但正式沉淀前仍需要质量提示。

## skill 候选

### 1. 收件箱候选沉淀

- 建议名称：`inbox-distill`
- 优先级：高
- 价值：把巡检建议真正变成候选草稿，形成“入箱 -> 分拣 -> 候选沉淀 -> 回链”的闭环。
- 适用目标：
  - `75_提示词库/`
  - `20_资料库/`
  - `30_原子笔记/`
  - `10_项目/个人数据资产系统/任务清单.md`
- 风险：不能自动确认长期知识，必须明确输出为候选草稿。

### 2. 提示词库整理

- 建议名称：`prompt-library-curator`
- 优先级：中
- 价值：把社媒文案、对话片段和项目经验整理为可复用提示词。
- 风险：容易把来源内容过度改写，需要保留“不是逐字稿”的声明。

### 3. 转写质量审计

- 建议名称：`transcript-quality-check`
- 优先级：中
- 价值：在沉淀前扫描 ASR 错词、英文术语、过短摘要和低质量时间线。
- 风险：规则容易变成硬编码，需要保持为可扩展词表和风险提示。

## 脚本候选

### 1. `distill_inbox_note.py`

- 优先级：高
- 输入：收件箱笔记路径、目标类型。
- 输出：候选草稿、来源回链、状态回写预览。
- 默认模式：`--dry-run`，只输出将要生成的内容。
- 写入模式：`--write`，生成候选草稿并把来源状态改为 `已分拣`。

### 2. `update_inbox_status.py`

- 优先级：中
- 输入：收件箱笔记路径、目标状态、回链。
- 输出：只更新 frontmatter 和 `沉淀记录`。
- 价值：把状态回写做成可验证小动作，避免每次手工改。

### 3. `audit_transcript_terms.py`

- 优先级：低到中
- 输入：收件箱笔记。
- 输出：疑似 ASR 错词、未归一化英文术语、需要人工校对的片段。

## 自动化候选

### 1. 定时巡检收件箱

- 优先级：高
- 状态：已经有自动化提案，适合继续试运行。
- 建议：先每天或每 3 小时只输出报告，不自动写候选草稿。

### 2. 高置信候选沉淀提醒

- 优先级：中
- 触发条件：巡检结果中出现 `75_提示词库/` 且风险少、内容短、转写已校正。
- 行为：提醒“这条适合沉淀为提示词候选”，不自动写文件。

### 3. 每周沉淀复盘

- 优先级：中
- 触发条件：每周固定时间。
- 行为：统计入箱数量、已分拣数量、已晋升数量、剩余知识债。

## 最建议先推进的一项

先推进 `inbox-distill` 的最小版本。

原因：

1. 已经有 `inbox-capture` 负责捕获。
2. 已经有 `inbox-triage` 负责建议。
3. 当前缺口正好是“把建议变成候选草稿并回链”。
4. 这一步低风险，因为输出仍是候选，不确认长期知识。
5. 它能直接减少用户高频触发后的整理负担。

## 落地计划

第一版只支持一个目标：从单条收件箱内容生成提示词库候选。

建议范围：

1. 新增 skill：`.codex/skills/inbox-distill/`
2. 新增脚本：`.codex/skills/inbox-distill/scripts/distill_inbox_note.py`
3. 支持参数：
   - `--source <收件箱笔记路径>`
   - `--target prompt`
   - `--dry-run`
   - `--write`
4. 输出文件：
   - `75_提示词库/<标题>.md`
5. 回写来源：
   - `处理状态: 已分拣`
   - 追加 `## 沉淀记录`
6. 禁止行为：
   - 不标记 `已晋升`
   - 不删除或移动来源
   - 不自动写入决策记录

## 验证方式

1. 用已处理过的短视频素材做 dry run，确认生成内容稳定。
2. 用 `--write` 处理一条测试副本，确认：
   - 生成提示词库候选。
   - 来源回链正确。
   - 巡检待分拣数量减少。
3. 运行：
   - `python3 -m py_compile`
   - `quick_validate.py`
   - `git diff --check`
   - `python3 .codex/skills/inbox-triage/scripts/triage_inbox.py`

## 风险和人工确认点

- 候选提示词不等于来源逐字稿，需要标注“整理版”。
- `tiny` 转写内容不能直接作为事实依据。
- 长视频不适合作为第一版 `inbox-distill` 的自动目标。
- `已分拣` 只代表候选沉淀完成，不代表长期知识确认。
- 是否晋升到 `30_原子笔记/` 或项目决策，仍需人工确认或项目应用验证。
