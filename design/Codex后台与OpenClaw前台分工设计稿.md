## 一句话定位

Codex 是 `my-mind` 的后台工程师和知识治理层，负责质量、判断、修复、分拣和沉淀；OpenClaw 是用户身边的前台秘书，负责接收、提醒、转述、追问和收集反馈。

这套分工的核心不是能力强弱，而是场景不同：

- Codex 更适合低频但复杂、需要工程判断和文件治理的任务。
- OpenClaw 更适合高频、短消息、移动端、提醒和反馈采集。

## 总体原则

### 1. 前台不做最终判断

OpenClaw 可以帮助用户把资料放进系统，也可以提醒用户该看什么，但不直接判断某条资料是否成为长期知识。

长期知识的晋升必须经过 Codex 后台处理，并在必要时获得用户明确反馈。

### 2. 后台不抢前台交互

Codex 不应该承担即时聊天秘书、移动端通知、每日提醒和高频追问。

Codex 应该把需要用户参与的事项整理成结构化推送，由 OpenClaw 转述给用户。

### 3. 入箱先保真，再分拣，再沉淀

资料进入系统后先通过入箱门禁，保证来源、标题、作者、链接、摘录、字幕或 OCR 等基础信息尽量可信。

只有入箱质量过关后，才进入分拣；只有用户阅读或确认后，才进入沉淀。

### 4. 自动化只做低风险动作

允许自动完成：

- 补齐明显缺失的元数据。
- 修正明显格式问题。
- 转繁体为简体。
- 标记解析失败或待补抓。
- 追加分拣记录。
- 生成阅读队列。
- 生成前台推送。

禁止自动完成：

- 删除资料。
- 覆盖原文。
- 确认长期知识。
- 把候选内容标记为已晋升。
- 直接修改关键项目决策。

### 5. 所有关键动作可追溯

入箱、门禁、分拣、推送、反馈、沉淀都应该留下文件记录。

至少要能回答：

- 这条资料从哪里来？
- 谁在什么时候处理过？
- 当前处理状态是什么？
- 用户有没有读过或反馈过？
- 如果沉淀了，沉淀到哪里？

### 6. 本地是源库，飞书是手机阅读层

`my-mind` 本地 Markdown 仓库是唯一可信源，负责版本、状态、回链、自动化记录和长期知识结构。

飞书知识库适合承担手机阅读镜像层：只同步精选待读、项目摘要、候选沉淀和需要用户反馈的内容，不做全量双向同步。

OpenClaw 通过飞书完成移动端触达和反馈采集，Codex 负责把反馈回流到本地知识库。

## 角色边界

### Codex 后台职责

#### 入箱质量门禁

目标：确保 `00_收件箱/` 不是一堆坏链接和半截内容。

Codex 应负责：

- 检查 frontmatter 是否完整。
- 检查来源链接是否保留。
- 检查标题、平台、作者、发布时间是否可信。
- 检查摘要、摘录、字幕、OCR 是否存在。
- 检查是否有重复入箱。
- 检查是否存在解析失败占位。
- 检查是否有繁体未转简体。
- 检查是否有明显术语误识别，例如把 `Codex` 识别成错误中文。
- 检查是否有 OCR、下载或转写残留进程。
- 对低风险异常直接修复。

当前已配置自动化：

- 名称：`收件箱入箱门禁审核`
- 频率：每 4 小时一次
- 运行者：Codex App 后台自动化
- 边界：只修入箱质量，不沉淀长期知识

#### 收件箱分拣

目标：把入箱资料变成可读、可判断、可后续沉淀的队列。

Codex 应负责：

- 扫描 `处理状态: 待分拣` 的条目。
- 判断优先级、建议去向和下一步动作。
- 自动把低风险条目标为 `已分拣`。
- 追加 `## 分拣记录`。
- 输出阅读队列。
- 明确告诉用户读完后要反馈什么。

当前已配置自动化：

- 名称：`收件箱待分拣巡检`
- 频率：每 3 小时一次
- 运行者：Codex App 后台自动化
- 边界：只做自动分拣和阅读队列，不生成候选沉淀

#### 反馈消费和回写

目标：把 OpenClaw 收集到的短反馈变成来源笔记里的结构化记忆。

Codex 应负责：

- 读取 `85_运行记录/前台反馈队列.jsonl`。
- 判断反馈对应哪条收件箱资料。
- 把 `已读`、`补充想法` 回写到来源笔记的 `## 阅读思考`。
- 对 `沉淀` 反馈生成候选沉淀物。
- 对 `继续解析` 反馈触发补抓、字幕、OCR 或重转写。
- 避免重复消费同一条反馈。

#### 候选沉淀

目标：把用户确认过的资料转为可复用资产。

Codex 应负责：

- 生成 `75_提示词库/` 候选。
- 生成 `20_资料库/` 候选。
- 生成 `30_原子笔记/` 候选。
- 生成项目任务、问题或决策候选。
- 在来源笔记追加回链和沉淀记录。
- 保留候选状态，等待用户确认。

#### 项目进度治理

目标：让 `my-mind` 不只是资料仓库，而是能持续推动项目的系统。

Codex 应负责：

- 巡检 `10_项目/个人数据资产系统/任务清单.md`。
- 检查设计稿、任务、运行记录之间是否脱节。
- 发现已经完成但未更新的任务。
- 发现长期卡住的事项。
- 给 OpenClaw 生成“项目进度摘要”，供其提醒用户。

#### 工具链维护

目标：保证入箱、解析、OCR、字幕、分拣、沉淀脚本长期可用。

Codex 应负责：

- 诊断平台解析失败。
- 优化 OCR 质量和性能。
- 调整字幕或转写策略。
- 修复 skill 脚本。
- 增加安全超时和进程清理。
- 维护 OpenClaw 可调用的稳定命令。

### OpenClaw 前台职责

#### 高频入箱

目标：让用户不需要想格式，直接把资料丢进系统。

OpenClaw 应负责：

- 接收用户发来的链接、文字、截图或语音备注。
- 调用 `inbox-capture` 写入 `00_收件箱/`。
- 返回入箱结果。
- 告诉用户是否解析成功。
- 如果解析失败，给出简短提醒，不自行深挖修复。

建议回复格式：

```text
已入箱：标题
状态：已解析 / 需后台补抓 / OCR 进行中 / 解析失败
路径：00_收件箱/...
```

#### 即时异常提醒

目标：用户丢资料时马上知道有没有问题。

OpenClaw 应提醒：

- 链接无法访问。
- 平台需要登录态。
- OCR 被跳过。
- 只有标题没有正文。
- 内容疑似重复。
- 后台门禁建议用户补充上下文。

OpenClaw 不负责修复杂问题，只负责把异常转成用户能理解的短消息。

#### 前台推送

目标：把 Codex 的后台判断变成用户愿意看的短提醒。

OpenClaw 应负责：

- 读取最新 `85_运行记录/前台推送-*.md`。
- 每次只推 1 到 3 个重点事项。
- 避免把完整巡检报告转发给用户。
- 用短句说明为什么值得看。
- 明确用户可以如何回复。

建议推送结构：

```text
今天最值得看：

1. 标题
为什么看：...
你可以回复：1 已读：想法 / 1 沉淀 / 1 跳过
```

#### 反馈收集

目标：让用户用很短的话完成系统状态推进。

OpenClaw 应收集：

- `已读`
- `补充想法`
- `沉淀`
- `跳过`
- `继续解析`
- `稍后提醒`

写入方式：

```bash
python3 .codex/skills/frontdesk-feedback/scripts/append_frontdesk_feedback.py --channel "<渠道>" "<用户回复>"
```

OpenClaw 只追加反馈队列，不直接改来源笔记。

#### 秘书式追问

目标：在不打扰用户的情况下，补齐沉淀所需的最小信息。

适合 OpenClaw 追问的问题：

- “这条你是想收藏，还是想沉淀成提示词？”
- “你读完后最想保留的一句话是什么？”
- “这条和哪个项目有关？”
- “需要我稍后再提醒你看吗？”

不适合 OpenClaw 追问的问题：

- “这条是否应该成为长期知识？”
- “这个项目的最终架构是什么？”
- “是否修改设计稿和任务清单？”

这些问题应交给 Codex 后台或用户主动决策。

#### 轻量状态播报

目标：让用户随时知道系统有没有堆积。

OpenClaw 可以播报：

- 今天新增几条收件箱。
- 有几条解析失败。
- 有几条待读。
- 有几条等待反馈。
- 有几条后台建议沉淀。
- 今天是否有项目进度提醒。

OpenClaw 不需要解释所有细节，只报告需要用户行动的部分。

### 飞书知识库手机阅读层

#### 定位

飞书知识库不是 `my-mind` 的替代品，而是面向手机阅读的前台镜像。

它解决的是阅读场景问题：

- 用户主要在手机上读资料。
- 用户不应该为了阅读打开本地仓库。
- 用户需要能在阅读后马上回复一句反馈。
- OpenClaw 已经通过飞书触达用户，适合承接提醒和反馈。

#### 推荐结构

第一版建议在飞书知识库中只维护一个轻量空间：

```text
my-mind 手机阅读
├── 今日待读
├── 收件箱精选
├── 项目推进
├── 候选沉淀
└── 已读归档
```

这些页面是阅读入口，不是源数据。

本地对应关系仍以 `00_收件箱/`、`85_运行记录/`、`10_项目/` 和候选沉淀目录为准。

#### 同步原则

- 只做本地到飞书的精选发布，不做全量同步。
- 飞书页面可以被覆盖或重新生成，本地 Markdown 不应依赖飞书作为唯一副本。
- 飞书页面必须保留本地来源路径和处理状态。
- 用户在飞书里的回复不直接改本地文件，先由 OpenClaw 写入反馈队列。
- Codex 消费反馈队列后，再回写 `阅读思考` 或触发候选沉淀。

#### 适合发布到飞书的内容

- 每日 1 到 3 条待读资料。
- Codex 生成的前台推送。
- 项目今日最小推进动作。
- 需要用户确认的候选沉淀。
- 入箱门禁发现、但需要用户判断的异常项。

#### 不适合发布到飞书的内容

- 全量 `00_收件箱/`。
- 全量 `20_资料库/` 或 `30_原子笔记/`。
- 包含隐私或敏感信息的原始资料。
- 未经筛选的大段转写、OCR 噪声和调试日志。
- 自动化内部状态和脚本运行细节。

#### 可用命令方向

本机已安装 `lark-cli`，后续可用它实现手机阅读发布：

```bash
lark-cli docs +create --api-version v2 --doc-format markdown --content @85_运行记录/前台推送-YYYY-MM-DD-HHMM.md --parent-position my_library
lark-cli wiki +node-create --title "今日待读" --space-id "<飞书知识库 space id>"
```

第一版不要求立刻做复杂双向同步，只需要做到：

1. Codex 生成手机阅读稿。
2. Codex 发布到飞书文档或知识库节点。
3. OpenClaw 把飞书链接推给用户。
4. 用户在飞书或 OpenClaw 对话中回复。
5. OpenClaw 把反馈写入本地反馈队列。

## 推荐新增任务

### Codex 后台可以继续承担的任务

#### 1. 反馈队列消费任务

目的：把 OpenClaw 收到的用户反馈真正写回系统。

建议频率：

- 初期：每天 1 次。
- 高频使用后：每 4 到 6 小时一次。

输出：

- 回写到来源笔记的 `## 阅读思考`。
- `85_运行记录/反馈消费-YYYY-MM-DD-HHMM.md`。

#### 2. 项目进度巡检任务

目的：检查设计稿、任务清单、运行记录是否同步。

建议频率：

- 每天 1 次轻巡检。
- 每周 1 次完整复盘。

输出：

- 当前完成事项。
- 卡住事项。
- 下一个最小动作。
- 需要 OpenClaw 提醒用户的事项。

#### 3. 知识库健康检查任务

目的：防止资料越积越乱。

建议频率：

- 每周 1 次。

检查内容：

- 重复标题。
- 已分拣但长期未读。
- 已处理但无回链。
- 候选沉淀长期未确认。
- 文件命名不一致。
- 状态字段异常。

#### 4. 解析器回归测试任务

目的：保证 YouTube、抖音、小红书、X 等入口持续可用。

建议频率：

- 每周 1 次。
- 解析失败集中出现时手动触发。

输出：

- 哪个平台解析正常。
- 哪个平台只能拿基础信息。
- 哪个平台需要登录态或人工补充。
- 哪个脚本需要修。

#### 5. 每周数据飞轮复盘

目的：看系统是否真的让资料变成资产。

建议频率：

- 每周 1 次。

指标：

- 本周入箱数量。
- 分拣数量。
- 阅读数量。
- 沉淀数量。
- 形成提示词、资料、原子笔记、项目任务的数量。
- 最大堵点。

#### 6. 飞书手机阅读发布任务

目的：把 Codex 后台整理出的待读和项目摘要发布到飞书，服务手机阅读场景。

建议频率：

- 初期：手动触发或每日 1 次。
- 稳定后：每天早上发布“今日待读”，晚上发布“轻复盘”。

输出：

- 飞书文档或知识库节点。
- 本地 `85_运行记录/飞书发布-YYYY-MM-DD-HHMM.md`。
- 可由 OpenClaw 转发给用户的飞书链接。

### OpenClaw 前台可以继续承担的任务

#### 1. 每日早间秘书播报

内容：

- 昨天新增了什么。
- 今天最值得读哪 1 到 3 条。
- 有哪些需要用户一句话反馈。
- 有没有解析失败需要补充。

#### 2. 晚间轻复盘

内容：

- 今天入箱了几条。
- 哪些还没读。
- 哪条值得明天优先看。
- 是否有一条可以沉淀。

#### 3. 读后追问

当用户回复 `已读` 后，OpenClaw 可以继续问一句：

```text
要不要把这条沉淀？可以回复：沉淀成提示词 / 沉淀成资料 / 先跳过。
```

#### 4. 异常短提醒

当 Codex 门禁报告出现人工判断项时，OpenClaw 只推最短动作：

```text
有 1 条小红书资料只有标题，没有正文。要我标记为待补抓，还是先跳过？
```

#### 5. 项目节奏提醒

OpenClaw 可以根据 Codex 生成的项目摘要提醒：

- 今天最小推进动作。
- 哪个任务卡住了。
- 哪个设计稿需要确认。
- 哪个候选沉淀等待用户拍板。

#### 6. 飞书阅读链接转发

OpenClaw 可以把 Codex 发布到飞书的阅读链接推给用户：

- 今天待读。
- 项目推进摘要。
- 候选沉淀确认页。
- 解析异常待确认页。

OpenClaw 的重点不是生成飞书内容，而是把正确链接在正确时间送到用户面前，并收集用户回复。

## 文件和权限边界

### OpenClaw 可写

```text
00_收件箱/
85_运行记录/前台反馈队列.jsonl
```

其中 `00_收件箱/` 应优先通过 `inbox-capture` 写入，不建议 OpenClaw 手写 Markdown。

### OpenClaw 可读

```text
00_收件箱/
85_运行记录/前台推送-*.md
85_运行记录/收件箱分拣巡检-*.md
85_运行记录/反馈消费-*.md
10_项目/个人数据资产系统/任务清单.md
```

### OpenClaw 不应直接写

```text
10_项目/
20_资料库/
30_原子笔记/
60_行业情报/
65_洞察/
75_提示词库/
design/
```

如需影响这些目录，应通过反馈队列交给 Codex 后台处理。

### Codex 可写

Codex 可以写全部项目目录，但长期知识和项目决策仍应遵守人工确认边界。

### 飞书写入边界

飞书知识库和飞书文档只作为发布层使用。

允许写入：

- `今日待读`
- `收件箱精选`
- `项目推进`
- `候选沉淀`
- `已读归档`

不建议写入：

- 未筛选的原始收件箱全集。
- 私密或敏感资料。
- 自动化调试日志。
- 尚未门禁通过的 OCR 或转写噪声。

飞书侧内容允许重建，本地 `my-mind` 仍是状态和知识的源头。

## 典型流程

### 流程一：用户丢一个链接

1. 用户把链接发给 OpenClaw。
2. OpenClaw 调用 `inbox-capture` 入箱。
3. OpenClaw 返回入箱结果。
4. Codex 的入箱门禁自动检查质量。
5. 如果有异常，Codex 修复低风险问题，并把人工判断项交给前台推送。
6. Codex 分拣后生成阅读队列。
7. OpenClaw 在合适时间提醒用户阅读。

### 流程二：用户读完一条资料

1. OpenClaw 推送待读事项。
2. 用户回复 `1 已读：我的想法...`。
3. OpenClaw 追加写入反馈队列。
4. Codex 消费反馈队列。
5. Codex 把想法回写到来源笔记的 `## 阅读思考`。
6. 如果用户明确回复 `沉淀`，Codex 生成候选沉淀物。

### 流程三：用户在手机上阅读

1. Codex 生成 `85_运行记录/前台推送-*.md`。
2. Codex 把精选内容发布到飞书文档或知识库节点。
3. OpenClaw 把飞书链接推给用户。
4. 用户在手机飞书里阅读。
5. 用户回复 OpenClaw：`1 已读：...`、`1 沉淀` 或 `1 跳过`。
6. OpenClaw 把反馈写入 `前台反馈队列.jsonl`。
7. Codex 回写本地来源笔记或进入候选沉淀流程。

### 流程四：解析失败

1. OpenClaw 入箱时发现解析失败。
2. OpenClaw 返回简短提示。
3. Codex 门禁审核发现异常。
4. Codex 尝试补抓、OCR、字幕或重解析。
5. 如果仍失败，Codex 记录原因。
6. OpenClaw 提醒用户是否补充截图、正文或放弃。

### 流程五：项目推进

1. Codex 后台巡检项目任务和运行记录。
2. Codex 生成项目进度摘要。
3. OpenClaw 每日或每周推送一句话下一步。
4. 用户回复是否执行、跳过或调整优先级。
5. Codex 根据反馈更新任务、设计或沉淀计划。

## 阶段计划

### 阶段一：稳定入箱和门禁

目标：让资料进入系统后不丢、不乱、不残。

已具备：

- `inbox-capture`
- `收件箱入箱门禁审核`
- 小红书 OCR 默认走 PP-OCRv5 文本 OCR
- 抖音、小红书、YouTube 基础信息采集

下一步：

- 跑一周观察门禁报告。
- 收集常见异常类型。
- 把高频异常固化进 `inbox-capture`。

### 阶段二：稳定前台秘书闭环

目标：OpenClaw 能完成“提醒用户读、收集用户反馈”的闭环。

需要完成：

- OpenClaw 读取最新 `前台推送-*.md`。
- Codex 把精选阅读稿发布到飞书文档或知识库节点。
- OpenClaw 把飞书阅读链接推送给用户。
- OpenClaw 按 1 到 3 条事项推给用户。
- OpenClaw 把回复写入 `前台反馈队列.jsonl`。
- Codex 定期消费反馈队列。

### 阶段三：半自动沉淀

目标：用户确认后，Codex 自动生成候选资产。

需要完成：

- 支持资料库候选。
- 支持原子笔记候选。
- 支持项目任务候选。
- 来源笔记自动追加回链。

### 阶段四：项目和知识库健康治理

目标：系统不仅能收资料，还能持续变好。

需要完成：

- 项目进度巡检自动化。
- 知识库健康检查自动化。
- 每周数据飞轮复盘。
- OpenClaw 每周秘书汇报。

## 验收标准

第一版跑通时，应满足：

- 用户可以只把链接丢给 OpenClaw，不关心 Markdown 格式。
- OpenClaw 能告诉用户入箱是否成功。
- Codex 能自动发现入箱质量问题。
- Codex 能把资料分拣成阅读队列。
- OpenClaw 能把阅读队列转成短提醒。
- 用户能用一句话反馈。
- Codex 能把反馈回写到来源笔记。
- 没有用户确认时，不会产生长期知识晋升。

## 风险和控制

### OpenClaw 权限过宽

风险：前台交互频繁，容易把噪声直接写入长期知识。

控制：OpenClaw 只写收件箱和反馈队列，不直接写长期知识目录。

### Codex 自动化过度沉淀

风险：后台把未经用户确认的资料变成看似确定的知识。

控制：分拣、门禁、沉淀拆开；沉淀必须有用户确认或明确任务触发。

### 解析内容质量不稳定

风险：社媒平台字段缺失、OCR 错词、字幕误识别。

控制：先门禁，后分拣；分拣报告必须暴露质量风险。

### 提醒过多

风险：OpenClaw 变成噪音源。

控制：前台推送每次最多 3 条，只推需要用户行动的事项。

### 飞书和本地状态冲突

风险：用户在飞书里编辑内容，导致飞书版本和本地 Markdown 版本不一致。

控制：第一版不做双向同步；飞书只作为阅读镜像和反馈入口，本地 `my-mind` 是唯一可信源。

### 状态不同步

风险：收件箱、运行记录、任务清单状态互相脱节。

控制：Codex 定期做项目进度巡检和知识库健康检查。

---

## 当前建议

当前最值得推进的不是让 OpenClaw 做更多复杂判断，而是先让它稳定做好三件事：

1. 入箱后即时告诉用户结果。
2. 每日读取 Codex 的前台推送，并把飞书阅读链接短提醒给用户。
3. 把用户一句话反馈写入反馈队列。

Codex 后台则继续补齐三件事：

1. 入箱门禁审核稳定运行。
2. 飞书手机阅读稿发布。
3. 反馈队列消费和阅读思考回写。
4. 项目进度巡检和每周复盘。

这会形成一个清晰闭环：

```text
用户丢资料 -> OpenClaw 入箱 -> Codex 门禁 -> Codex 分拣 -> Codex 生成推送 -> Codex 发布飞书阅读页 -> OpenClaw 提醒 -> 用户手机阅读并反馈 -> OpenClaw 入队 -> Codex 回写或沉淀
```
