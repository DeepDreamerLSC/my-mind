## 一句话定位

Codex 是 `my-mind` 的后台工程师和知识治理层，负责质量、判断、修复、分拣和沉淀；OpenClaw 是用户身边的前台秘书，负责接收、提醒、转述、追问和收集反馈。

这套分工的核心不是能力强弱，而是场景不同：

- Codex 更适合低频但复杂、需要工程判断和文件治理的任务。
- OpenClaw 更适合高频、短消息、移动端、提醒和反馈采集。

## 总体原则

### 1. 前台不做最终判断

OpenClaw 可以帮助用户把资料放进系统，也可以提醒用户该看什么，但不直接判断某条资料是否成为长期知识。

长期知识的晋升必须经过 Codex 后台处理，并在必要时获得用户明确反馈。

### 2. 后台不抢前台交互

Codex 不应该承担即时聊天秘书、移动端通知、每日提醒和高频追问。

Codex 应该把需要用户参与的事项整理成结构化推送，由 OpenClaw 转述给用户。

### 3. 入箱先保真，再分拣，再沉淀

资料进入系统后先通过入箱门禁，保证来源、标题、作者、链接、摘录、字幕或 OCR 等基础信息尽量可信。

只有入箱质量过关后，才进入分拣；只有用户阅读或确认后，才进入沉淀。

### 4. 自动化只做低风险动作

允许自动完成：

- 补齐明显缺失的元数据。
- 修正明显格式问题。
- 转繁体为简体。
- 标记解析失败或待补抓。
- 追加分拣记录。
- 生成阅读队列。
- 生成前台推送。

禁止自动完成：

- 删除资料。
- 覆盖原文。
- 确认长期知识。
- 把候选内容标记为已晋升。
- 直接修改关键项目决策。

### 5. 所有关键动作可追溯

入箱、门禁、分拣、推送、反馈、沉淀都应该留下文件记录。

至少要能回答：

- 这条资料从哪里来？
- 谁在什么时候处理过？
- 当前处理状态是什么？
- 用户有没有读过或反馈过？
- 如果沉淀了，沉淀到哪里？

### 6. 本地是源库，飞书是手机阅读层

`my-mind` 本地 Markdown 仓库是唯一可信源，负责版本、状态、回链、自动化记录和长期知识结构。

飞书知识库适合承担手机阅读镜像层：只同步精选待读、项目摘要、候选沉淀和需要用户反馈的内容，不做全量双向同步。

OpenClaw 通过飞书完成移动端触达和反馈采集，Codex 负责把反馈回流到本地知识库。

## 角色边界

### Codex 后台职责

#### 入箱质量门禁

目标：确保 `00_收件箱/` 不是一堆坏链接和半截内容。

Codex 应负责：

- 检查 frontmatter 是否完整。
- 检查来源链接是否保留。
- 检查标题、平台、作者、发布时间是否可信。
- 检查摘要、摘录、字幕、OCR 是否存在。
- 检查是否有重复入箱。
- 检查是否存在解析失败占位。
- 检查是否有繁体未转简体。
- 检查是否有明显术语误识别，例如把 `Codex` 识别成错误中文。
- 检查是否有 OCR、下载或转写残留进程。
- 维护 `内容质量` 和 `质量门禁` 字段，阻止低质量内容直接进入前台推送。
- 对低风险异常直接修复。

当前已配置自动化：

- 名称：`收件箱入箱门禁审核`
- 频率：每 4 小时一次
- 运行者：Codex App 后台自动化
- 边界：只修入箱质量，不沉淀长期知识

#### 收件箱分拣

目标：把入箱资料变成可读、可判断、可后续沉淀的队列。

Codex 应负责：

- 扫描 `处理状态: 待分拣` 的条目。
- 判断优先级、建议去向和下一步动作。
- 自动把低风险条目标为 `已分拣`。
- 追加 `## 分拣记录`。
- 输出阅读队列。
- 明确告诉用户读完后要反馈什么。

当前已配置自动化：

- 名称：`收件箱待分拣巡检`
- 频率：每 6 小时一次
- 运行者：Codex App 后台自动化
- 边界：只做自动分拣和阅读队列，不生成候选沉淀

#### 反馈消费和回写

目标：把 OpenClaw 收集到的短反馈变成来源笔记里的结构化记忆。

Codex 应负责：

- 读取 `85_运行记录/前台反馈队列.jsonl`。
- 判断反馈对应哪条收件箱资料。
- 把 `已读`、`补充想法` 回写到来源笔记的 `## 阅读思考`。
- 对 `沉淀` 反馈生成候选沉淀物。
- 对 `继续解析` 反馈触发补抓、字幕、OCR 或重转写。
- 避免重复消费同一条反馈。

#### 候选沉淀

目标：把用户确认过的资料转为可复用资产。

Codex 应负责：

- 生成 `75_提示词库/` 候选。
- 生成 `20_资料库/` 候选。
- 生成 `30_原子笔记/` 候选。
- 生成项目任务、问题或决策候选。
- 在来源笔记追加回链和沉淀记录。
- 保留候选状态，等待用户确认。

#### 项目进度治理

目标：让 `my-mind` 不只是资料仓库，而是能持续推动项目的系统。

Codex 应负责：

- 巡检 `10_项目/个人数据资产系统/任务清单.md`。
- 检查设计稿、任务、运行记录之间是否脱节。
- 发现已经完成但未更新的任务。
- 发现长期卡住的事项。
- 给 OpenClaw 生成“项目进度摘要”，供其提醒用户。

#### 工具链维护

目标：保证入箱、解析、OCR、字幕、分拣、沉淀脚本长期可用。

Codex 应负责：

- 诊断平台解析失败。
- 优化 OCR 质量和性能。
- 调整字幕或转写策略。
- 修复 skill 脚本。
- 增加安全超时和进程清理。
- 维护 OpenClaw 可调用的稳定命令。

### OpenClaw 前台职责

#### 高频入箱

目标：让用户不需要想格式，直接把资料丢进系统。

OpenClaw 应负责：

- 接收用户发来的链接、文字、截图或语音备注。
- 调用 `inbox-capture` 写入 `00_收件箱/`。
- 用户主动发来的链接默认写入 `阅读状态: 已读`；自动发现或用户明确未读的链接才写入 `阅读状态: 未读`。
- 返回入箱结果。
- 告诉用户是否解析成功。
- 对抖音、公开视频等视频链接，默认调用普通入箱命令即可触发可访问短视频转写；前提是公开页面或 `yt-dlp` 能暴露媒体/音频地址，且时长未超过上限。普通入箱默认上限是 360 秒；用户明确说“入库”时，`knowledge-intake` 默认把上限提高到 3600 秒，也就是 1 小时。同时区分“基础信息已解析”和“视频内容已转写”。如果 `内容摘录来源` 为空，必须提示需后台继续解析，而不是说“无法转写”。
- 如果解析失败，给出简短提醒，不自行深挖修复。

#### 主动入库

目标：用户明确说“入库”时，OpenClaw 不只做入箱，而是触发受控候选沉淀。

OpenClaw 应负责：

- 识别 `入库`、`入到知识库`、`保存到知识库`、`直接沉淀` 等明确意图。
- 调用 `.codex/skills/knowledge-intake/scripts/knowledge_intake.py --write --raw "<用户原始消息>"`。
- 如果用户指定已有收件箱文件，改用 `--source "00_收件箱/..."`。
- 只返回原始材料路径、候选结果、入库处理报告和最小待确认问题。
- 不手写 `20_资料库/`、`65_洞察/`、`75_提示词库/` 等长期目录。
- 不替用户确认候选晋升、飞书同步、删除或归档。

本机 OpenClaw workspace 已有薄入口 `my-mind-knowledge-intake`；它不复制逻辑，只委托 my-mind 仓库内脚本。

建议回复格式：

```text
已入箱：标题
状态：已解析 / 需后台补抓 / OCR 进行中 / 解析失败
内容：已生成摘录 / 未转写，需后台继续解析
路径：00_收件箱/...
```

#### 即时异常提醒

目标：用户丢资料时马上知道有没有问题。

OpenClaw 应提醒：

- 链接无法访问。
- 平台需要登录态。
- OCR 被跳过。
- 只有标题没有正文。
- 内容疑似重复。
- 后台门禁建议用户补充上下文。

OpenClaw 不负责修复杂问题，只负责把异常转成用户能理解的短消息。

#### 前台推送

目标：把 Codex 的后台判断变成用户愿意看的短提醒。

OpenClaw 应负责：

- 调用 `.codex/skills/feishu-publish/scripts/build_openclaw_feishu_message.py` 生成最终聊天消息。
- 如果没有与最新 `85_运行记录/前台推送-*.md` 匹配的已发布飞书记录，先触发飞书发布，不退回发送原文链接。
- 聊天消息只推飞书链接、少量重点标题和回复格式；完整内容放在飞书页。
- 避免把完整巡检报告转发给用户。
- 用短句说明为什么值得看。
- 明确用户可以如何回复。

建议推送结构：

```text
今天最值得看：

1. 标题
为什么看：...
你可以回复：1 已读：想法 / 1 沉淀 / 1 跳过
```

#### 反馈收集

目标：让用户用很短的话完成系统状态推进。

OpenClaw 应收集：

- `已读`
- `补充想法`
- `沉淀`
- `跳过`
- `继续解析`
- `稍后提醒`

写入方式：

```bash
python3 .codex/skills/frontdesk-feedback/scripts/append_frontdesk_feedback.py --channel "<渠道>" "<用户回复>"
```

OpenClaw 只追加反馈队列，不直接改来源笔记。

#### 秘书式追问

目标：在不打扰用户的情况下，补齐沉淀所需的最小信息。

适合 OpenClaw 追问的问题：

- “这条你是想收藏，还是想沉淀成提示词？”
- “你读完后最想保留的一句话是什么？”
- “这条和哪个项目有关？”
- “需要我稍后再提醒你看吗？”

不适合 OpenClaw 追问的问题：

- “这条是否应该成为长期知识？”
- “这个项目的最终架构是什么？”
- “是否修改设计稿和任务清单？”

这些问题应交给 Codex 后台或用户主动决策。

#### 轻量状态播报

目标：让用户随时知道系统有没有堆积。

OpenClaw 可以播报：

- 今天新增几条收件箱。
- 有几条解析失败。
- 有几条待读。
- 有几条等待反馈。
- 有几条后台建议沉淀。
- 今天是否有项目进度提醒。

OpenClaw 不需要解释所有细节，只报告需要用户行动的部分。

### 飞书知识库手机阅读层

#### 定位

飞书知识库不是 `my-mind` 的替代品，而是面向手机阅读的前台镜像。

它解决的是阅读场景问题：

- 用户主要在手机上读资料。
- 用户不应该为了阅读打开本地仓库。
- 用户需要能在阅读后马上回复一句反馈。
- OpenClaw 已经通过飞书触达用户，适合承接提醒和反馈。

#### 推荐结构

第一版建议在飞书知识库中维护三类入口：

1. 手机待读入口，承接每日或阶段性阅读页。
2. 待沉淀入口，承接已读、已入库或已生成候选但仍需确认的内容。
3. 精选知识入口，承接已经本地归类、适合手机回看的长期内容。

飞书目录只保持本地知识库的“可读骨架”，不复制本地所有工作目录：

```text
林总工的知识库
├── 📱 my-mind 手机待读
│   ├── 今日待读
│   ├── 待读归档
│   └── 反馈入口说明
├── 🧪 my-mind 待沉淀
│   ├── 候选资料
│   ├── 候选提示词
│   └── 待确认问题
├── 10_项目精选
│   └── 个人数据资产系统
├── 20_资料库精选
│   ├── 人工智能产业
│   ├── AI产品与工具
│   ├── 工作流与自动化
│   ├── 管理与组织
│   ├── 设计与视觉
│   ├── 写作与表达
│   └── 人文与社会
├── 75_提示词库精选
└── 🧹 已纳入本地待清理
```

这些页面是阅读入口，不是源数据。飞书目录名可以更适合手机阅读，但语义必须能映射回本地目录。

本地对应关系仍以 `00_收件箱/`、`05_流转区/`、`85_运行记录/`、`10_项目/` 和候选沉淀目录为准。

#### 同步原则

- 只做本地到飞书的单向发布，不做全量双向同步。
- 目录骨架保持一致，内容只同步精选。
- 飞书页面可以被覆盖或重新生成，本地 Markdown 不应依赖飞书作为唯一副本。
- 飞书页面必须保留本地来源路径和处理状态。
- 用户在飞书里的回复不直接改本地文件，先由 OpenClaw 写入反馈队列。
- Codex 消费反馈队列后，再回写 `阅读思考` 或触发候选沉淀。

#### 内容分层

| 层级 | 本地来源 | 飞书形态 | 同步策略 |
| --- | --- | --- | --- |
| 手机待读 | `85_运行记录/前台推送-*.md`、`85_运行记录/飞书精选页/` | `📱 my-mind 手机待读` 下的精选索引页和单篇文章 | 可更新当天精选索引，单篇文章按来源复用 |
| 待沉淀 | `05_流转区/30_待沉淀/`、`85_运行记录/入库处理-*.md`、候选文件 | `🧪 my-mind 待沉淀` 下的候选摘要和问题清单 | 只作为确认入口，不等同长期知识 |
| 项目精选 | `10_项目/个人数据资产系统/项目总览.md`、`任务清单.md`、关键设计稿 | `10_项目精选/` | 只同步用户手机上需要看的摘要和下一步 |
| 资料精选 | `20_资料库/` 中已归类、具备复用价值的资料 | `20_资料库精选/` 对应主题目录 | 只同步标记为精选的资料 |
| 提示词精选 | `75_提示词库/` 中稳定可复用提示词 | `75_提示词库精选/` | 只同步经过试用的版本 |
| 运行证据 | `85_运行记录/` | 通常不同步 | 只保留本地审计；必要时生成摘要页 |
| 原始收件箱 | `00_收件箱/` | 通常不同步 | 只有被分拣为精选后才进入资料或阅读页 |

#### 同步状态字段

需要进入飞书精选知识库的本地文档，建议在 front matter 中加入同步字段。第一版只要求最小字段，不把飞书空间 ID 或凭证写入仓库：

```yaml
飞书同步:
  策略: 精选同步
  状态: 待同步
  飞书页面: ""
  最近同步: ""
  内容哈希: ""
```

字段含义：

- `策略`：`不同步`、`目录页同步`、`精选同步`、`手机待读发布`。
- `状态`：`待同步`、`已同步`、`需更新`、`暂停`、`失败`。
- `飞书页面`：用户可打开的飞书文档或知识库链接。
- `最近同步`：最近一次同步时间。
- `内容哈希`：用于判断本地内容是否变化，避免重复创建页面。

同步执行结果写入：

```text
85_运行记录/飞书知识库同步记录.jsonl
```

该记录只保存运行证据、飞书链接、页面 token、内容 hash 和错误信息，不保存凭证。

#### 适合发布到飞书的内容

- 当前全量待读资料。
- Codex 生成的前台推送。
- `knowledge-intake` 生成的候选摘要和最小确认问题。
- 项目今日最小推进动作。
- 需要用户确认的候选沉淀。
- 入箱门禁发现、但需要用户判断的异常项。
- 已经归类、值得复读或引用的 `20_资料库/` 精选资料。
- 经用户确认可复用的提示词、方法论、设计稿和项目摘要。

#### 不适合发布到飞书的内容

- 全量 `00_收件箱/`。
- 全量 `20_资料库/` 或 `30_原子笔记/`。
- 包含隐私或敏感信息的原始资料。
- 未经筛选的大段转写、OCR 噪声和调试日志。
- 自动化内部状态和脚本运行细节。

#### 可用命令方向

已落地第一版 `feishu-publish` skill。当前推荐发布为飞书精选 bundle：先为每条待读生成单篇文章，再生成一个精选索引页；真实发布通过命令模板接入。为了让用户能在手机飞书知识库中看到页面，发布命令必须使用 OpenClaw workspace 的用户身份，而不是 bot-only 身份：

```bash
OPENCLAW_HOME="$HOME/.openclaw" lark-cli config strict-mode user
OPENCLAW_HOME="$HOME/.openclaw" lark-cli config default-as user
OPENCLAW_HOME="$HOME/.openclaw" lark-cli auth status --verify
```

确认 `identity=user` 后再发布：

```bash
python3 .codex/skills/feishu-publish/scripts/publish_frontdesk_bundle.py --dry-run
MY_MIND_FEISHU_PUBLISH_COMMAND='OPENCLAW_HOME="$HOME/.openclaw" lark-cli docs +create --api-version v2 --wiki-space my_library --title {title} --content @{markdown_file_rel}' \
MY_MIND_FEISHU_WIKI_SPACE_ID='目标知识库space_id' \
MY_MIND_FEISHU_WIKI_PARENT_NODE_TOKEN='手机待读目录node_token' \
python3 .codex/skills/feishu-publish/scripts/publish_frontdesk_bundle.py --publish
```

若目标是手机飞书知识库目录，而不是普通云文档搜索结果，则发布后自动移动到目标知识库。空间 ID 用本机环境变量注入，不写进仓库：

```bash
export MY_MIND_FEISHU_WIKI_SPACE_ID='目标知识库space_id'
export MY_MIND_FEISHU_WIKI_PARENT_NODE_TOKEN='目标目录node_token'
MY_MIND_FEISHU_PUBLISH_COMMAND='OPENCLAW_HOME="$HOME/.openclaw" lark-cli docs +create --api-version v2 --wiki-space my_library --title {title} --content @{markdown_file_rel}' \
python3 .codex/skills/feishu-publish/scripts/publish_frontdesk_bundle.py --publish
```

第一版不要求立刻做复杂双向同步。需要先完成两条轻量链路：

手机待读发布链路：

1. Codex 生成手机阅读稿。
2. Codex 发布飞书精选 bundle：每条待读生成一篇单篇文章，并生成一个精选索引页。
3. Codex 追加 `85_运行记录/飞书发布记录.jsonl`。
4. OpenClaw 调用飞书消息出口脚本，把飞书知识库精选索引链接推给用户。
5. 用户在飞书或 OpenClaw 对话中回复。
6. OpenClaw 把反馈写入本地反馈队列。

入库处理链路：

1. 用户对 OpenClaw/Codex 说“入库”。
2. `knowledge-intake` 先调用采集能力保存到 `00_收件箱/`。
3. `knowledge-intake` 复用分拣能力，只回写本次来源。
4. 能安全候选化的内容生成到 `20_资料库/`、`75_提示词库/` 或 `65_洞察/候选洞察/`，状态保持候选/待确认。
5. 不安全的内容进入待核验或确认问题清单。
6. OpenClaw 只把最小问题清单推给用户，避免暴露全部后台机械步骤。

长期精选同步链路：

1. Codex 扫描带有 `飞书同步` 字段的本地文档。
2. Codex 根据本地目录映射到飞书精选目录。
3. Codex 用内容 hash 判断是创建、更新还是跳过。
4. Codex 把同步结果写入 `85_运行记录/飞书知识库同步记录.jsonl`。
5. OpenClaw 只读取最新同步结果或飞书链接，不负责判断哪些内容应该入选精选。

## 推荐新增任务

### Codex 后台可以继续承担的任务

#### 1. 反馈队列消费任务

目的：把 OpenClaw 收到的用户反馈真正写回系统。

建议频率：

- 初期：每天 1 次。
- 高频使用后：每 4 到 6 小时一次。

输出：

- 回写到来源笔记的 `## 阅读思考`。
- `85_运行记录/反馈消费-YYYY-MM-DD-HHMM.md`。

#### 2. 项目进度巡检任务

目的：检查设计稿、任务清单、运行记录是否同步。

建议频率：

- 每天 1 次轻巡检。
- 每周 1 次完整复盘。

输出：

- 当前完成事项。
- 卡住事项。
- 下一个最小动作。
- 需要 OpenClaw 提醒用户的事项。

#### 3. 知识库健康检查任务

目的：防止资料越积越乱。

建议频率：

- 每周 1 次。

检查内容：

- 重复标题。
- 已分拣但长期未读。
- 已处理但无回链。
- 候选沉淀长期未确认。
- 文件命名不一致。
- 状态字段异常。

#### 4. 解析器回归测试任务

目的：保证 YouTube、抖音、小红书、X 等入口持续可用。

建议频率：

- 每周 1 次。
- 解析失败集中出现时手动触发。

输出：

- 哪个平台解析正常。
- 哪个平台只能拿基础信息。
- 哪个平台需要登录态或人工补充。
- 哪个脚本需要修。

#### 5. 每周数据飞轮复盘

目的：看系统是否真的让资料变成资产。

建议频率：

- 每周 1 次。

指标：

- 本周入箱数量。
- 分拣数量。
- 阅读数量。
- 沉淀数量。
- 形成提示词、资料、原子笔记、项目任务的数量。
- 最大堵点。

#### 6. 飞书手机阅读发布任务

目的：把 Codex 后台整理出的待读和项目摘要发布到飞书，服务手机阅读场景。

建议频率：

- 初期：手动触发或每日 1 次。
- 稳定后：每天早上发布“今日待读”，晚上发布“轻复盘”。

输出：

- 飞书文档或知识库节点。
- 本地 `85_运行记录/飞书发布-YYYY-MM-DD-HHMM.md`。
- 可由 OpenClaw 转发给用户的飞书链接。

#### 7. 飞书知识库精选同步任务

目的：把本地已经归类、值得复读的资料、项目摘要和提示词同步到飞书知识库，让手机端有一个干净的精选阅读库。

建议频率：

- 初期：手动触发，每次只同步少量标记文档。
- 稳定后：每周 1 次同步精选内容，日常仍以手机待读页为主。

输入：

- 带 `飞书同步` 字段且策略为 `目录页同步` 或 `精选同步` 的本地 Markdown。
- 本地目录到飞书目录的映射规则。
- 上一次 `85_运行记录/飞书知识库同步记录.jsonl`。

输出：

- 飞书知识库对应目录下的精选页面。
- 本地文档中的飞书页面链接和同步状态。
- `85_运行记录/飞书知识库同步记录.jsonl`。

不做：

- 不同步全量收件箱、全量运行记录和未筛选转写。
- 不把飞书侧编辑自动回写到本地。
- 不让 OpenClaw 判断精选入选标准。

### OpenClaw 前台可以继续承担的任务

#### 1. 每日早间秘书播报

内容：


```text
有 1 条小红书资料只有标题，没有正文。要我标记为待补抓，还是先跳过？
```

OpenClaw 的重点不是生成飞书内容，而是把正确链接在正确时间送到用户面前，并收集用户回复。

## 文件和权限边界

### OpenClaw 可写

```text
00_收件箱/
85_运行记录/前台反馈队列.jsonl
```

其中 `00_收件箱/` 应优先通过 `inbox-capture` 写入，不建议 OpenClaw 手写 Markdown。

### OpenClaw 可读

```text
00_收件箱/
05_流转区/
85_运行记录/前台推送-*.md
85_运行记录/收件箱分拣巡检-*.md
85_运行记录/反馈消费-*.md
10_项目/个人数据资产系统/任务清单.md
```

### OpenClaw 不应直接写

```text
10_项目/
20_资料库/
30_原子笔记/
60_行业情报/
65_洞察/
75_提示词库/
design/
```

如需影响这些目录，应通过反馈队列交给 Codex 后台处理。

### Codex 可写

Codex 可以写全部项目目录，但长期知识和项目决策仍应遵守人工确认边界。

### 飞书写入边界

飞书知识库和飞书文档只作为发布层使用。

允许写入：

- `📱 my-mind 手机待读`
- `10_项目精选`
- `20_资料库精选`
- `75_提示词库精选`
- `🧹 已纳入本地待清理`

不建议写入：

- 未筛选的原始收件箱全集。
- 私密或敏感资料。
- 自动化调试日志。
- 尚未门禁通过的 OCR 或转写噪声。
- 本地还没有归类或没有同步标记的普通资料。

飞书侧内容允许重建，本地 `my-mind` 仍是状态和知识的源头。

## 典型流程

### 流程一：用户丢一个链接

1. 用户把链接发给 OpenClaw。
2. OpenClaw 调用 `inbox-capture` 入箱。
3. OpenClaw 返回入箱结果。
4. Codex 的入箱门禁自动检查质量。
5. 如果有异常，Codex 修复低风险问题，并把人工判断项交给前台推送。
6. Codex 分拣后生成阅读队列。
7. OpenClaw 在合适时间提醒用户阅读。

### 流程二：用户读完一条资料

1. OpenClaw 推送待读事项。
2. 用户回复 `1 已读：我的想法...`。
3. OpenClaw 追加写入反馈队列。
4. Codex 消费反馈队列。
5. Codex 把想法回写到来源笔记的 `## 阅读思考`。
6. 如果用户明确回复 `沉淀`，Codex 生成候选沉淀物。

### 流程三：用户在手机上阅读

1. Codex 生成 `85_运行记录/前台推送-*.md`。
2. Codex 把精选内容发布成飞书单篇文章，并生成精选索引页。
3. OpenClaw 调用飞书消息出口脚本，把飞书精选索引链接推给用户。
4. 用户在手机飞书里阅读。
5. 用户回复 OpenClaw：`1 已读：...`、`1 沉淀` 或 `1 跳过`。
6. OpenClaw 把反馈写入 `前台反馈队列.jsonl`。
7. Codex 回写本地来源笔记或进入候选沉淀流程。

### 流程四：解析失败

1. OpenClaw 入箱时发现解析失败。
2. OpenClaw 返回简短提示。
3. Codex 门禁审核发现异常。
4. Codex 尝试补抓、OCR、字幕或重解析。
5. 如果仍失败，Codex 记录原因。
6. OpenClaw 提醒用户是否补充截图、正文或放弃。

### 流程五：项目推进

1. Codex 后台巡检项目任务和运行记录。
2. Codex 生成项目进度摘要。
3. OpenClaw 每日或每周推送一句话下一步。
4. 用户回复是否执行、跳过或调整优先级。
5. Codex 根据反馈更新任务、设计或沉淀计划。

## 阶段计划

### 阶段一：稳定入箱和门禁

目标：让资料进入系统后不丢、不乱、不残。

已具备：

- `inbox-capture`
- `收件箱入箱门禁审核`
- 小红书 OCR 默认走 PP-OCRv5 文本 OCR
- 抖音、小红书、YouTube 基础信息采集

下一步：

- 跑一周观察门禁报告。

- 把高频异常固化进 `inbox-capture`。

## 风险和控制

### OpenClaw 权限过宽

风险：前台交互频繁，容易把噪声直接写入长期知识。

控制：OpenClaw 只写收件箱和反馈队列，不直接写长期知识目录。

### Codex 自动化过度沉淀

风险：后台把未经用户确认的资料变成看似确定的知识。

控制：分拣、门禁、沉淀拆开；沉淀必须有用户确认或明确任务触发。

### 解析内容质量不稳定

风险：社媒平台字段缺失、OCR 错词、字幕误识别。

控制：先门禁，后分拣；分拣报告必须暴露质量风险。

### 提醒过多

风险：OpenClaw 变成噪音源。

控制：飞书页可以承载全量待读，但 OpenClaw 聊天消息只发链接、少量重点标题和回复格式。

### 飞书和本地状态冲突

风险：用户在飞书里编辑内容，导致飞书版本和本地 Markdown 版本不一致。

控制：第一版不做双向同步；飞书只作为阅读镜像和反馈入口，本地 `my-mind` 是唯一可信源。

### 飞书知识库膨胀

风险：把本地目录全量同步到飞书后，手机端也变成噪声仓库。

控制：飞书只保持目录骨架一致，内容必须经过精选标记；`00_收件箱/`、`05_流转区/` 和 `85_运行记录/` 默认不同步。

### 状态不同步

风险：收件箱、运行记录、任务清单状态互相脱节。

控制：Codex 定期做项目进度巡检和知识库健康检查。

---

## 当前建议

当前最值得推进的不是让 OpenClaw 做更多复杂判断，而是先让它稳定做好三件事：

1. 入箱后即时告诉用户结果。
2. 每日读取 Codex 的前台推送，并把飞书阅读链接短提醒给用户。
3. 把用户一句话反馈写入反馈队列。

Codex 后台则继续补齐三件事：

1. 入箱门禁审核稳定运行。
2. 飞书精选 bundle 发布。
3. 反馈队列消费和阅读思考回写。
4. 项目进度巡检和每周复盘。
5. 后台总控巡检和前台推送节流。

这会形成一个清晰闭环：

```text
用户丢资料 -> OpenClaw 入箱 -> Codex 门禁 -> Codex 分拣 -> Codex 生成推送 -> Codex 发布飞书精选 bundle -> OpenClaw 提醒 -> 用户手机阅读并反馈 -> OpenClaw 入队 -> Codex 回写或沉淀
```
