## 一句话定位

收件箱自动分拣不是自动替你决定长期知识，而是先把 `00_收件箱/` 里的待分拣内容自动分拣成“已分拣 + 流转区队列”。如果用户只是“入箱”，系统停在保存和分拣；如果用户明确说“入库”，系统仍先入箱保存证据链，然后尽量自动生成候选知识和最小确认问题。

## 三层内容模型

### 待读内容

还只是值得看一眼的输入候选。来源主要是 `00_收件箱/`、`05_流转区/10_待读/` 和前沿情报巡检。它不代表认可，也不代表要沉淀。

动作：阅读、跳过、继续解析、补一句判断。

### 待沉淀内容

已经读过、用户主动入库，或系统已生成候选草稿，但还没有变成长期知识。来源主要是 `05_流转区/30_待沉淀/`、`85_运行记录/入库处理-*.md` 和来源笔记的 `沉淀记录`。

动作：确认沉淀到资料库、提示词库、候选洞察或项目；必要时补核验和改写。

### 化为己有的内容

已经进入 `20_资料库/`、`30_原子笔记/`、`65_洞察/`、`75_提示词库/` 等长期目录，并且有来源、回链、结构化标题、我的理解或使用记录。飞书里对应的是精选知识库镜像，不是临时待读页。

动作：复用、同步、复盘、持续修订。

## 设计目标

1. 降低高频入箱后的整理负担。
2. 让每条资料在被遗忘前获得一次结构化判断。
3. 保持低风险自动化：先分析和建议，不直接晋升长期知识。
4. 为后续自动生成资料库草稿、原子笔记候选、提示词候选和项目任务候选打基础。
5. 保留来源可追溯、人工确认和敏感信息检查。

## 非目标

- 不在定时分拣阶段自动移动文件、改写正文内容或生成候选沉淀草稿。
- 用户明确说“入库”时，允许 `knowledge-intake` 生成候选草稿，但候选仍标记为待确认。
- 第一阶段只允许低风险回写，例如 `处理状态`、`关联项目` 和 `分拣记录`。
- 不自动把候选内容标记为长期知识。
- 不自动确认事实、结论、决策或项目方向。
- 不绕过平台访问控制，不重新抓取需要登录态的内容。

## 核心对象

### 输入

- `00_收件箱/` 中 `处理状态: 待分拣` 的笔记。
- 重点字段：标题、来源平台、作者、发布时间、解析状态、内容摘录来源、内容摘录后端、内容摘录字数、标签、正文摘录。

### 输出

- 中文分拣报告。
- `05_流转区/` 中的短期行动视图。
- 每条资料包含：
  - 建议优先级
  - 核心价值
  - 建议去向
  - 下一步动作
  - 需要人工处理的风险

### 建议去向

- `20_资料库/`：值得保留为资料，但还没有拆成长期知识。
- `30_原子笔记/`：能形成长期可复用知识点。
- `75_提示词库/`：包含可复用提示词、Codex 工作流、skill 或自动化方法。
- `10_项目/个人数据资产系统/`：会影响当前系统设计、任务、问题或决策。
- `60_行业情报/`：属于外部趋势、产品、公司或技术变化信号。
- `65_洞察/候选洞察/`：跨资料可形成判断，但仍需验证。
- `保持待分拣`：信息不足、价值不明或需要补抓。
- `可丢弃`：重复、低价值或无法解析且没有保留意义。

### 流转区

新增 `05_流转区/`，作为收件箱和长期知识之间的短期行动面板：

- `05_流转区/10_待读/`：已分拣、等待用户阅读和反馈的条目。
- `05_流转区/30_待沉淀/`：已有阅读反馈或候选草稿，等待继续沉淀或确认的条目。
- `05_流转区/40_待核验/`：存在解析、转写、来源或事实风险，正式沉淀前需要校对的条目。
- `05_流转区/90_暂缓/`：暂时不处理、但不想立刻丢弃的条目。

流转区只保存队列和链接，不移动原始资料。收件箱仍是来源事实，长期知识仍进入资料库、原子笔记、行业情报、洞察、提示词库或项目目录。流转区队列不是互斥状态，同一条资料可以同时出现在“待读”和“待核验”等多个视图中。

### 手机阅读发布

收件箱链路不在入箱瞬间直接发布飞书，避免坏链接、半截解析和低价值内容打扰用户。推荐串联点是“已分拣并进入流转区”之后：

```text
入箱 -> 分拣 -> 流转区待读 -> 前台推送 -> 飞书精选 bundle -> OpenClaw 索引链接提醒 -> 用户反馈 -> Codex 回写或沉淀
```

当前由 `feishu-publish` 读取最新 `85_运行记录/前台推送-*.md`，生成 `85_运行记录/飞书精选页/单篇/*.md` 和 `85_运行记录/飞书精选页/索引/*.md`，并维护 `85_运行记录/飞书发布记录.jsonl`。真实飞书创建通过命令模板接入，不把凭证、space id 或租户细节写入仓库；发布身份应使用用户态，避免 bot-only 创建后手机知识库不可见。

### 入库处理

`knowledge-intake` 是“入库”的上层入口：

```text
入库 -> 先入箱 -> 分拣 -> 生成候选资料/提示词/洞察 -> 刷新流转区 -> 输出确认问题
```

它复用 `inbox-capture` 的解析、OCR 和转写能力，不复制底层采集逻辑；复用 `inbox-triage` 的分类能力，但只回写本次入库涉及的来源，避免一次入库影响全部待分拣条目。

默认安全规则：

- 新材料必须先保存到 `00_收件箱/`。
- `内容质量: 需继续解析` 或 `解析状态: 解析失败` 时，不生成候选。
- 自动发现或未读材料默认停在待读，除非显式 `--force-unread`。
- 候选文件写入长期目录时仍标记 `处理状态: 候选`、`吸收状态: 待确认`。
- 候选资料即使位于 `20_资料库/`，也默认写入 `飞书同步: 策略: 不同步 / 状态: 暂停`，避免未确认内容进入飞书精选知识库。
- 来源笔记追加 `沉淀记录` 回链，但不标记为已晋升。

## 分阶段方案

### 阶段一：巡检和分拣建议

目标：让 Codex 能定期或手动扫描收件箱，自动完成低风险分拣，并生成阅读队列。

能力：

1. 读取 `00_收件箱/*.md`。
2. 识别 `处理状态: 待分拣`。
3. 对明显可分拣的条目自动回写 `处理状态: 已分拣`，追加 `## 分拣记录`。
4. 汇总当前 `已分拣` 条目，生成阅读队列。
5. 基于元数据和正文关键词给出优先级与建议去向。
6. 标记需要人工校对的风险，例如：
   - `tiny` 模型转写错词。
   - 社媒公开页字段不完整。
   - 官方转录存在但尚未整理。
   - 内容过长，需要先抽重点。
7. 输出巡检报告到终端，必要时写入 `85_运行记录/`。
8. 写报告时同步刷新 `05_流转区/当前流转总览.md`、待读队列、待沉淀队列和待核验队列。

验收标准：

- 能对当前收件箱生成报告。
- 只做低风险回写：允许把条目标为 `已分拣` 并追加 `分拣记录`，不生成候选沉淀草稿。
- 报告能指出每条资料的下一步。
- 能明确列出转写质量、事实核验和晋升风险。
- 报告能给出当前待读队列，并提示用户读完后主动反馈。
- `05_流转区/` 能提供当前行动视图，且不复制长篇原文。
- 用户反馈后，可把反馈回写到来源笔记的 `## 阅读思考` 章节，再进入下一步候选沉淀。

### 阶段二：候选沉淀草稿

目标：在用户阅读确认，或用户主动说“入库”时，把高价值条目生成为候选草稿。

当前试运行状态：

- 第一版已完成 `75_提示词库/` 候选提示词生成。
- 已验证 `00_收件箱/2026-06-10 小红书 - Codex Agent 工作流优化.md` 可生成 `75_提示词库/Codex工作流/Codex Agent 工作流优化.md`。
- 写入后来源笔记会标记为 `处理状态: 已处理`，并追加 `## 沉淀记录` 回链。
- 新增 `knowledge-intake` 作为通用入库入口，可生成 `20_资料库/`、`75_提示词库/` 和 `65_洞察/候选洞察/` 候选；默认只把来源保持在 `已分拣 + 已生成候选`，等待确认。

能力：

1. 生成 `20_资料库/` 资料草稿。
2. 生成 `30_原子笔记/` 候选原子笔记。
3. 生成 `75_提示词库/` 可复用提示词候选。
4. 生成 `10_项目/个人数据资产系统/任务清单.md` 的任务候选。
5. 在来源笔记中只追加回链或处理记录，不覆盖原文。

状态回写规则：

- 用户主动发来的入箱链接默认写入 `阅读状态: 已读`，因为这类链接通常是用户已经读过、看过或至少判断过值得保存的内容。
- 自动发现、巡检抓取或用户明确表示尚未阅读的链接，才写入 `阅读状态: 未读`。
- `阅读状态: 已读` 的条目分拣后进入“待沉淀/待补判断”视图，不再作为普通待读材料反复提醒。
- 自动分拣完成后，来源笔记改为 `处理状态: 已分拣`，并追加 `## 分拣记录`。
- 用户阅读后，可先把反馈回写到来源笔记的 `## 阅读思考` 章节。
- 用户明确要求继续沉淀后，可以生成资料库、原子笔记、提示词库或项目候选草稿，并追加 `## 沉淀记录` 回链。
- 如果是 `knowledge-intake` 自动候选，来源笔记保持 `处理状态: 已分拣`，并增加 `入库状态: 已生成候选`，使它继续出现在待沉淀确认队列。
- 如果是用户明确确认“这条已经处理完”，来源笔记才改为 `处理状态: 已处理`。
- 只有候选内容经过人工确认或项目应用后，才可把来源笔记标为 `处理状态: 已晋升`。
- 不因为自动转写、自动摘要或候选草稿生成而自动确认长期知识。

人工确认点：

- 是否现在继续沉淀。
- 是否先把交流后的判断写回 `阅读思考`。
- 是否接受摘要。
- 是否确认原子笔记。
- 是否把任务加入项目。

- 类型：Codex App `cron` 自动化
- 运行环境：`local`
- 工作目录：`/Users/linsuchang/Desktop/work/my-mind`

- 推理强度：`high`
- 调度：`FREQ=DAILY;BYHOUR=7;BYMINUTE=10;BYSECOND=0`，即每天 07:10 一次

当前自动化 prompt 边界：

1. 第一步只处理 `处理状态: 待分拣` 的条目。
2. 允许把来源回写为 `已分拣` 并追加 `分拣记录`。

输出方式：

- 同步写入 `85_运行记录/收件箱分拣巡检-日期.md`。

  - “反馈可回写到 `阅读思考`”的明确提醒

### 阶段四：半自动沉淀闭环

目标：用户只需确认批次，Codex 自动生成草稿、回链和复盘指标。

仍然禁止：

- 自动确认长期知识。
- 自动删除资料。
- 自动公开同步敏感内容。

### 阶段五：后台总控和推送节流

目标：避免自动化线程各自运行导致重复推送、状态漂移或质量问题扩散。

已落地能力：

1. 新增 `backend-control` skill，读取自动化配置、流转区、收件箱质量、反馈队列、前台推送状态、飞书同步记录和 git 状态，输出后台健康报告。
2. 前台推送新增 `85_运行记录/前台推送状态.json`，记录每条资料的 `source_file`、`content_hash`、`last_pushed_at`、`feedback_status` 和 `push_count`。
3. 前台推送默认跳过 24 小时内已推送但尚未反馈的条目，避免 OpenClaw 反复打扰用户；需要强制生成时使用 `--force`。
4. 入箱新增 `内容质量` 和 `质量门禁` 字段，取值为 `可推送`、`需核验`、`需继续解析`；前台推送默认跳过 `需继续解析`。
5. `frontdesk-feedback` 新增反馈消费脚本，可把 `已读` 和 `补充想法` 回写到来源笔记的 `阅读思考`，并在用户明确要求时生成候选提示词。

验收标准：

- 后台巡检能指出当前是否应该先消费反馈、补解析、推送或沉淀。
- 同一条资料未收到反馈前不会在短周期内重复推送。
- 低质量入箱资料不会默认进入手机阅读推送。
- 反馈消费有报告和状态回写，避免重复处理同一条反馈。

## 阶段一规则

### 优先级判断

- 高：直接服务当前项目、Codex 工作流、收件箱能力、skill、自动化或提示词库。
- 中：有长期资料价值，但需要先整理或验证。
- 低：解析不足、价值不明、重复或仅作备查。

### 风险判断

- 转写风险：`内容摘录后端: faster-whisper` 且模型为 `tiny`，或正文出现明显 ASR 误识别。
- 来源风险：只有社媒公开页，缺少完整原文或官方转录。
- 事实风险：涉及公司、产品、价格、发布日期、模型能力、市场判断等可能变化的信息。
- 沉淀风险：当前内容只适合做候选，不适合直接进入原子笔记或决策。

# 收件箱分拣巡检

## 试运行策略

第一轮只处理当前 `00_收件箱/` 中已经存在的待分拣内容，生成报告后人工查看：

1. 如果建议方向基本正确，接入定时巡检。
2. 如果建议方向偏差大，先调整关键词和评分规则。
3. 如果某类内容频繁出现，追加专门规则，例如 Codex 视频、提示词、行业访谈、产品更新、论文。
