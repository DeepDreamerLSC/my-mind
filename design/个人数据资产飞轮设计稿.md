## 一句话定位

我的心智不是“资料收藏夹”，而是一个以 Obsidian 本地知识库为底座、由 Codex 和本地脚本辅助运行的个人数据资产飞轮：持续收集资料，吸收知识，挖掘洞察，推动项目，产生输出，再用复盘结果优化下一轮判断。

## 对伙伴稿的审视

### 值得保留的核心判断

1. 原始资料必须保留，人工智能助手和脚本只能追加结构化信息，不能破坏来源。
2. 人工智能对话是重要资料源，但默认低信任，不能直接晋升为长期知识。
3. 知识库的目标不是“存得多”，而是“记住、用到、输出、影响决策”。
4. 洞察层应独立于原子笔记层，承载跨资料、跨项目、跨复盘的高阶判断。
5. 行业情报先作为信号进入系统，经过聚类、验证和人工确认后再影响项目。
6. Codex 的价值不只是整理文件，还包括上下文维护、缺口分析、复盘审计和自动化执行。

### 已做的优化

1. 收窄第一版范围：把原稿中的完整蓝图拆成“最小可用版本”和后续阶段，避免一开始就背上过多目录和流程。
2. 统一中文术语：目录名、文件名、状态、数据类型、模板字段和流程名称都改成中文表达。
3. 补全生命周期：明确资料如何从“已捕获”逐步晋升为原子笔记、洞察、任务、决策或输出。
4. 增加验收标准：每个阶段都定义了做完以后怎样算真的跑通。
5. 增加退场机制：低价值资料、重复资料、错误结论、敏感信息、过期洞察都要能被标记、归档或淘汰。
6. 保留人工确认：人工智能助手的整理结果默认是候选内容，长期知识和关键决策必须经过人工确认或项目验证。

## 设计原则

### 1. 原始内容不可破坏

所有网页、人工智能对话、播客转写、视频字幕、手写笔记、项目讨论和行业信号先进入原始资料层。系统可以追加摘要、标签、链接、可信度、任务和洞察，但不直接覆盖原文。

### 2. 自动化只负责低风险动作

Codex 和脚本可以做导入、分类建议、摘要、候选原子笔记、候选任务、候选洞察、指标统计。涉及长期观点、关键决策、项目方向、敏感信息处理时，必须保留人工确认状态。

### 3. 知识必须能被追溯

每条原子笔记、洞察、决策和输出候选都要能回链到来源。没有来源的内容只能是草稿，不能进入已确认知识层。

### 4. 吸收优先于收藏

系统的核心指标不是新增资料数量，而是吸收率、应用率、输出率、洞察转化率和项目影响率。

### 5. 项目是知识的试金石

知识只有进入项目上下文、影响任务、改变决策或生成输出，才算真正产生价值。

## 目标用户和使用场景

### 主要用户

- 项目推进者：需要把资料、人工智能对话、想法和行动统一管理。
- 学习者：需要把输入转化为可回忆、可应用、可讲清楚的知识。
- 创作者：需要从长期积累中持续产出文章、方案、脚本、研究报告和方法论。
- 人工智能协作者：Codex、ChatGPT、OpenClaw 等需要读取稳定上下文并执行整理、分析、审计任务。

### 核心任务

1. 快速收集一条资料，并保留来源。
2. 从人工智能对话中提炼项目上下文、决策、任务、原子笔记和待验证事实。
3. 把资料晋升为原子笔记、主动回忆卡片、洞察或项目行动。
4. 每周检查知识库是否真的推动了项目和输出。
5. 定期发现高频主题、矛盾观点、知识缺口、机会点和输出候选。

## 非目标

第一版不追求：

- 全自动知识管理。
- 大而全的目录一次性搭建。
- 完整替代任务管理器、客户关系管理、阅读器或发布系统。
- 让人工智能助手直接决定长期观点或项目方向。
- 为每一条低价值资料都生成完整模板。

## 核心数据类型

| 数据类型 | 用途 | 是否需要来源 |
| --- | --- | --- |
| 资料 | 文章、论文、播客、视频、报告、网页剪藏 | 是 |
| 人工智能对话 | ChatGPT、OpenClaw、Codex 等对话记录 | 是 |
| 项目 | 当前推进中的目标型工作 | 否 |
| 原子笔记 | 一个长期可复用的知识点 | 是 |
| 回忆卡片 | 主动回忆和间隔复习 | 是 |
| 行业信号 | 外部变化、新工具、新论文、新产品 | 是 |
| 洞察 | 多份资料交叉分析后的判断 | 是 |
| 决策 | 已选择的项目或系统方向 | 是 |
| 输出成果 | 文章、方案、报告、脚本、方法手册 | 建议 |
| 复盘 | 周复盘、月复盘、项目复盘 | 否 |

### 推荐状态枚举

```text
处理状态：已捕获 -> 已分拣 -> 已处理 -> 已晋升 -> 已归档
可信状态：原始 -> 有来源 -> 人工确认 -> 已应用 -> 已失效
吸收状态：已收集 -> 已理解 -> 已回忆 -> 已应用 -> 已讲清楚 -> 已内化
洞察状态：候选 -> 已验证 -> 已应用 -> 已退役
敏感状态：未知 -> 不敏感 -> 敏感 -> 禁止公开
```

目录、文件名、正文标题和元数据字段统一使用中文。Obsidian、Codex、OpenClaw、Dataview、Bases、Anki 等属于工具或产品专名，保留原名。

## 核心工作流

### 工作流 1：收件箱整理

输入：

- 临时想法
- 网页剪藏
- 论文或文档
- 播客或视频转写
- 人工智能对话导出
- 行业新闻或工具链接

处理规则：

1. 保留原始内容。
2. 补全基础元数据。
3. 标记资料类型、主题、关联项目、可信状态和处理状态。
4. 只生成候选摘要、候选任务、候选原子笔记，不直接确认长期知识。
5. 对敏感信息、未验证事实和重复资料做显式标记。

产出：

- 资料笔记
- 待验证事实
- 候选原子笔记
- 候选任务
- 处理报告

### 工作流 2：人工智能对话萃取

人工智能对话的默认定位是“低信任、高上下文密度资料”。

萃取目标：

- 项目上下文变化
- 已出现的决策
- 可执行任务
- 原子笔记候选
- 可复用提示词
- 待验证事实
- 敏感信息标记

晋升规则：

```text
原始对话 -> 对话笔记
对话笔记 -> 项目上下文候选
对话笔记 -> 决策候选
对话笔记 -> 任务候选
对话笔记 -> 原子笔记候选
对话笔记 -> 提示词候选
```

只有经过人工确认或项目应用后的内容，才能进入已确认决策、长期原子笔记或已验证洞察。

### 工作流 3：知识吸收

每条高价值原子笔记至少要能生成一个主动回忆问题。

吸收动作：

1. 先尝试回忆，不直接看答案。
2. 记录回忆评分和误区。
3. 设置下一次复习日期。
4. 找到至少一个项目应用场景。
5. 能讲清楚后再提升吸收状态。

### 工作流 4：洞察挖掘

洞察不是单条资料摘要，而是多个来源之间的模式。

系统定期寻找：

- 高频主题
- 重复问题
- 矛盾观点
- 项目知识缺口
- 行业趋势
- 机会点
- 输出候选
- 个人工作模式

每条洞察必须包含：

- 证据来源
- 可信度
- 影响项目
- 可行动性
- 下一步验证动作
- 过期或复盘日期

### 工作流 5：每周数据飞轮复盘

每周固定生成一次复盘：

1. 本周新增了什么。
2. 哪些资料被处理了。
3. 哪些知识被真正吸收或应用了。
4. 哪些洞察影响了项目。
5. 哪些内容可以转化为输出。
6. 哪些资料、信号或任务构成知识债。
7. 下周最重要的 3 个行动是什么。

## 风险和约束

### 范围膨胀

风险：目录和模板太多，导致系统启动成本过高。

处理：第一版只保留必要目录，其余放到后续阶段。

### 人工智能误判

风险：人工智能助手的摘要、分类、洞察存在错误。

处理：人工智能助手产出默认是候选内容，长期知识和关键决策需要人工确认或项目验证。

### 知识债堆积

风险：收集速度远高于处理和吸收速度。

处理：每周复盘必须输出知识债清单，并允许淘汰低价值资料源。

### 隐私泄露

风险：人工智能对话和项目资料可能包含敏感信息。

处理：所有导入资料必须有 `敏感状态` 字段；公开输出前必须审计。

### 过度自动化

风险：系统为了自动化而复杂化。

处理：先跑通人工加半自动闭环，再决定脚本和 Dataview/Bases 视图。

## 项目纲领总结

我的心智的第一目标不是搭一个漂亮的大目录，而是建立一个能持续运转的小闭环：

```text
收集资料
-> 保留来源
-> 结构化处理
-> 生成候选知识
-> 主动回忆和项目应用
-> 形成洞察
-> 推动任务或决策
-> 复盘有效性
-> 调整下一轮收集和分析
```

只要这个闭环能稳定运行，再扩展领域地图、输出成果库、行业情报聚类、自动化脚本和可视化仪表盘。否则目录越完整，系统越容易变成新的知识债。
