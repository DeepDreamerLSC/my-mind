# 决策记录

## 已确认决策

- 2026-06-09：第一版先落地最小结构，不一次性创建完整蓝图。
  - 依据：完整目录会提高启动成本，先跑通小闭环更重要。
  - 影响：阶段一只创建必要目录、项目骨架、核心模板和核心提示词。

- 2026-06-09：用户可见术语统一中文化。
  - 依据：用户明确要求相关术语都改成中文，包括文件名。
  - 影响：目录名、文件名、状态、数据类型、元数据字段和流程名称使用中文。

- 2026-06-09：人工智能对话是资料源，不是直接知识。
  - 依据：人工智能对话包含有效建议、未验证事实、临时上下文和可能错误推断。
  - 影响：对话先进入资料库，再萃取项目上下文、决策候选、任务候选、原子笔记候选和待验证事实。

- 2026-06-09：自动化只负责低风险动作。
  - 依据：关键观点、项目方向和敏感信息处理需要人工确认。
  - 影响：脚本和 Codex 先承担导入、整理、摘要、候选提炼、统计和审计。

- 2026-06-10：收件箱沉淀自动化先从巡检报告开始，不直接改写或晋升资料。
  - 依据：用户预计该流程会高频触发，但长期知识、项目任务和提示词沉淀仍需要先观察建议质量。
  - 影响：阶段一新增 `inbox-triage`，只读取待分拣笔记并输出建议；阶段二再考虑生成候选草稿。

- 2026-06-11：收件箱流程拆成“自动分拣”和“用户阅读后再沉淀”两个阶段。
  - 依据：用户希望 Codex 先自动分拣，但真正沉淀必须在用户看过资料并给出反馈后再进行。
  - 影响：`已分拣` 表示已进入待读队列；`已处理` 表示用户已确认继续沉淀并已生成候选草稿；自动化需要主动提示当前可阅读条目，而不是直接催沉淀。

- 2026-06-11：收件箱资料模板增加“阅读思考”，并把它作为读后反馈的正式落点。
  - 依据：用户希望把阅读后的判断、与 Codex 交流后的结论、认同与质疑点保存在原始资料旁边，而不是只停留在对话里。
  - 影响：资料模板、收件箱录入模板、阅读队列提示和候选沉淀输入都要包含 `阅读思考`；用户读后反馈可先回写，再决定是否继续沉淀。

- 2026-06-11：待沉淀巡检不只列条目，还要输出建议处理策略。
  - 依据：仅提示“还有哪些已分拣条目”不足以支持后续处理，用户需要直接看到当前阶段、已有候选和下一步处理方式。
  - 影响：`inbox-triage` 的待沉淀队列需要补充当前阶段、已有候选和建议处理策略，避免自动化结果停留在简单提醒。

- 2026-06-11：收件箱巡检已接入真实定时自动化，并保持“只分拣、不抢先沉淀”的执行边界。
  - 依据：收件箱分拣属于高频、低风险、重复性动作，适合先进入 Codex App `cron` 自动化试运行；但未读先沉淀会放大误判和噪声。
  - 影响：已启用 `收件箱待分拣巡检` 自动化，当前按每 3 小时一次运行；自动化允许回写 `已分拣` 和 `分拣记录`，并输出 `已分拣` 阅读队列，但在用户明确“已阅读”或“继续沉淀”前禁止生成资料库、原子笔记、提示词库或项目草稿。

- 2026-06-12：新增 `05_流转区/` 作为收件箱和长期知识之间的短期行动面板。
  - 依据：收件箱会持续膨胀，直接在 `00_收件箱/` 内阅读和筛选会越来越困难；但移动原始资料会破坏来源保真和现有脚本路径假设。
  - 影响：`00_收件箱/` 继续保存原始捕获，`05_流转区/` 只保存可自动刷新的待读、待沉淀、待核验和暂缓队列；长期知识仍进入资料库、原子笔记、行业情报、洞察、提示词库或项目目录。
